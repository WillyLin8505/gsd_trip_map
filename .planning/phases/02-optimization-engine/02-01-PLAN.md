## Phase Goal

**As a** developer calling the optimization engine, **I want to** order a set of resolved places by shortest travel time while respecting each place's opening hours, **so that** the engine produces a feasible, non-crossing single-route sequence that downstream day-splitting can build on.

  <action>
    Create `src/lib/optimizer/route.ts`. Implement three pure functions operating purely on integer indices and the `TravelMatrix` from `@/lib/optimizer/types`:
    - `routeTravelTime(order: number[], matrix: TravelMatrix): number` — sum of matrix[order[k]][order[k+1]] for consecutive pairs (open path, no return-to-origin leg).
    - `nearestNeighbor(matrix: TravelMatrix, startIndex = 0): number[]` — classic greedy: start at startIndex, repeatedly append the unvisited index j minimizing matrix[current][j], until all N indices are placed. O(N²). Returns an index permutation. This function orders by travel-time distance only (SCHED-03); opening-hours feasibility is layered in by the scheduler in Plan 02, not here.
    - `twoOptImprove(order: number[], matrix: TravelMatrix, passes = 4): number[]` — standard 2-opt: for `passes` iterations, for every pair (i, k) with i<k, reverse the segment order[i..k] and accept the reversal only if it lowers routeTravelTime; stop early if a full pass makes no improvement. Return the improved permutation. Default passes=4 (CONTEXT.md guidance is 3-5).

    Write failing tests FIRST in `src/lib/optimizer/route.test.ts` covering every behavior case, then implement until green. Keep these functions matrix-only and index-only — no place objects, no opening-hours logic — to preserve testability and separation of concerns.
