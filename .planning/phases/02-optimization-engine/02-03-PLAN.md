<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/optimize | Untrusted request body (placeIds, numDays, travelDate) crosses into the server; validated by Zod before any DB/Routes use. |
| server → Routes API | Server key + place coordinates leave to Google; key must stay server-side only. |
| server → places DB | placeIds used as a parameterized `inArray` filter; never interpolated as raw SQL. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-06 | Tampering | placeIds request param | high | mitigate | Zod schema validates placeIds is a non-empty string[]; Drizzle `inArray` parameterizes the query; non-cached placeIds rejected with 422 before any external call. |
| T-02-07 | Denial of Service / cost | unbounded placeIds → large Routes matrix | high | mitigate | Cap placeIds length (max 25, matching Routes API ≤25 waypoints / 625 elements per CONTEXT.md) in the Zod schema; reject oversized requests with 400 before calling Routes API. |
| T-02-08 | Information Disclosure | GOOGLE_PLACES_API_KEY | high | mitigate | Key read only from process.env in the Route Handler; never returned in a response; never passed to the client; routes-client receives it as an explicit server-side argument. |
| T-02-09 | Spoofing/Repudiation | silent drop of unresolved placeId | medium | mitigate | Unresolved placeIds returned explicitly in the 422 details list rather than dropped, so the caller cannot be misled about which places were scheduled. |

No package installs in this plan — Routes API uses the existing `fetch` and the Phase-1-installed stack; no new dependency, so no T-02-SC supply-chain surface.
</threat_model>
