<objective>
## Phase Goal (MVP user story)

**As a** traveller who has just generated a multi-day itinerary, **I want to** paste a new place on the results page and have it added to the geographically closest day, **so that** I can refine the plan without re-running the whole optimisation.

## This plan (Slice F1 — EDIT-01)

Deliver the complete F1 vertical slice end-to-end: a user pastes a place → it resolves → lands in the closest existing day (re-timed, order preserved) → renders in the browser. Failures never wipe the itinerary.

Purpose: First user-reachable capability of Phase 6 — single-day add without full re-optimisation.
Output: `pickClosestDay` + `scheduleSingleDay(reorder=false)` pure primitives, `POST /api/optimize/day` route, `DayPlaceAdder` UI + `replaceDay` client wiring.

Note: `scheduleSingleDay`'s `reorder=true` branch (meal-slotting) and `classifyPlace` are EDIT-02 scope, implemented in plan 06-02 which extends `src/lib/optimizer/day.ts`. F1 only ever calls the route with `reorder:false`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/optimize/day | Untrusted JSON (placeIds, reorder, dayNumber, travelDate) crosses into the server route |
| route → Google Routes API | Server-side computeRouteMatrix call; API key must stay server-only |
| route → Postgres | placeIds used in a parameterised inArray query |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-01 | Denial of Service | POST /api/optimize/day (oversized placeIds → cost) | medium | mitigate | optimizeDayRequestSchema caps placeIds at .max(25) (mirrors existing optimize route T-02-07); single computeRouteMatrix per request |
| T-06-02 | Information Disclosure | GOOGLE_PLACES_API_KEY | high | mitigate | Key read only from process.env in the route handler; never returned in any response or sent to the client (DayPlaceAdder calls our own routes only) |
| T-06-03 | Tampering | placeIds → DB query | medium | mitigate | inArray(places.place_id, placeIds) via Drizzle (parameterised); no raw SQL |
| T-06-04 | Denial of Service | Routes API spend (each F1 add triggers computeRouteMatrix) | low | accept | Capped at 25 placeIds; existing rate-limit on /api/places/resolve applies; no extra control for v1 |
</threat_model>
