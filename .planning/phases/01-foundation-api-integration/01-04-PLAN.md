<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-supplied URL → server redirect follow | Untrusted URL is fetched server-side (SSRF surface) |
| server → Google Places API | Cost-bearing calls gated by quota cap |
| operator → GCP Console | Manual config of the hard cost ceiling |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-13 | Denial of Service / cost | unbounded Google API spend | high | mitigate | Hard daily Places API quota cap + billing alerts at $10/$50/$100, verified active by the blocking human checkpoint before user-facing features ship |
| T-01-14 | Spoofing / SSRF | user-supplied Maps URL redirect follow | medium | mitigate | Only the expanded URL's /@lat,lng coordinates are used (fed to Google Text Search); response body is not fetched or rendered; coordinate extraction is the sole use of the redirect target |
| T-01-15 | Information Disclosure | GOOGLE_PLACES_API_KEY in url-resolver | high | mitigate | Key passed from the server route only; url-resolver is server-side; never imported by client components |
| T-01-16 | Denial of Service | malformed/looping redirect URLs | low | accept | fetch redirect "follow" caps redirect chains by default; a malformed URL yields null (NOT_FOUND) rather than a crash; per-user rate limiting that would further bound this is scoped to Phase 5 |
</threat_model>
