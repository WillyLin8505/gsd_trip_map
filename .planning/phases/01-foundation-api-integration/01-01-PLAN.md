## Phase Goal

**As a** trip planner, **I want to** paste a Chinese place name with a destination city and get back the resolved place (name, address, coordinates), **so that** I have a verified place I can later schedule into an itinerary.

<objective>
Walking Skeleton: scaffold the Next.js 15 + TypeScript + Tailwind v4 + Drizzle + Supabase stack and prove it works end-to-end with the thinnest real vertical slice — a UI form that submits a Chinese place name + destination city, a Route Handler that calls Google Places API (New) Text Search with mandatory field masking and city locationBias, a real upsert into the shared `places` cache table, and the resolved result rendered back in the UI.

This plan establishes the architectural backbone (per CONTEXT.md decisions: Next.js App Router, Drizzle ORM, Supabase, Places API New only, field masking on every call, two-key model). Subsequent plans (02 full schema + auth + RLS, 03 details/cache, 04 URL resolution + cost controls) build on this without renegotiating these choices.

Purpose: De-risk the full stack early — one real DB write, one real Google API call, one real UI interaction — and lock the directory layout in SKELETON.md.
Output: A runnable app, the `places` table, the resolve endpoint, the resolver form, and SKELETON.md.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form → /api/places/resolve | Untrusted user input (place names, city) crosses into the server |
| server route → Google Places API | Server key leaves the trust boundary; field mask controls cost/data |
| server route → Supabase DB | Upsert writes derived data into the shared cache |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Information Disclosure | GOOGLE_PLACES_API_KEY / SUPABASE_SERVICE_ROLE_KEY | high | mitigate | Keys read only in server route via process.env; never imported by client components; .env.example documents server-only keys. Acceptance criteria assert no client import. |
| T-01-02 | Tampering | resolve request body | medium | mitigate | Zod validation rejects malformed inputs and missing city with a 400 before any Google call or DB write |
| T-01-03 | Denial of Service / cost | Google Places Text Search | high | mitigate | Mandatory X-Goog-FieldMask (Essentials SKU only) on every call; hard GCP daily quota cap delivered in Plan 04 |
| T-01-04 | Spoofing | Chinese name ambiguity | medium | mitigate | locationBias (destination city) applied on every Text Search; resolved address returned for user confirmation |
| T-01-SC | Tampering | npm installs (Next.js, Drizzle, Supabase, Zod) | high | accept | All packages are first-party, well-known, and pre-approved in CLAUDE.md Technology Stack; no [ASSUMED]/[SUS] packages introduced. No package legitimacy audit table required since no novel/unaudited packages are added. |
